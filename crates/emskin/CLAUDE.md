# emskin - Nested Wayland Compositor for Emacs

## Build
- `cargo check` / `cargo clippy -- -D warnings` / `cargo fmt`
- smithay: forked at `loyalpartner/smithay` branch `emskin-patches`. Upstream: `Smithay/smithay`
- smithay patches: `backend/winit/mod.rs` (expose `WinitEvent::Ime`, 8-bit pixel format priority), `text_input/text_input_handle.rs` (remove `has_instance()` guard, add `cursor_rectangle` accessor), `selection/seat_data.rs` (fix GTK3 clipboard on focus change)

## Architecture
- Nested Wayland compositor using smithay, hosting Emacs inside a winit window
- First toplevel = Emacs (fullscreen), subsequent toplevels = **arbitrary embedded programs** (any Wayland or XWayland client) managed by AppManager. Not limited to EAF â€” any GTK/Qt/Electron/X11 app can be embedded as a child window whose geometry is controlled by Emacs via IPC.
- IPC protocol: length-prefixed JSON over Unix socket. Emacsâ†’compositor: set_geometry, close, set_visibility, prefix_done, set_focus, set_measure, add_mirror, update_mirror_geometry, remove_mirror, promote_mirror. Compositorâ†’Emacs: connected, surface_size, window_created, window_destroyed, title_changed, focus_view, xwayland_ready
- Elisp client: split across `elisp/emskin.el` (entry + shared state + user defvars), `emskin-ipc.el` (codec + connection + `emskin-connected-hook` + `emskin--jbool` helper), `emskin-app.el` (app lifecycle + geometry + mirrors + dispatch), `emskin-workspace.el` (workspace CRUD + frame mapping), and one file per effect toggle: `emskin-measure.el`, `emskin-skeleton.el`, `emskin-cursor-trail.el`, `emskin-jelly.el`. Each effect module registers its `--sync` helper on `emskin-connected-hook` at load time; on IPC connect, dispatch fires the hook so every effect pushes its current variable value. `emskin-apply-config` lets users re-fire the hook after `setq`. Auto-connects via parent PID socket discovery. All files are embedded into the binary via `include_dir!` and extracted at runtime in standalone mode
- Mirror system: same embedded program displays in multiple Emacs windows. Source = first window (real surface), mirrors = subsequent windows (TextureRenderElement from same GPU texture). Elisp tracks source/mirror in `emskin--mirror-table`
- Keyboard input: compositor detects Emacs prefix keys (C-x, C-c, M-x) via `input_intercept`, redirects focus to Emacs; `prefix_done` IPC restores focus. `set_focus` IPC for explicit focus control. Prefix state: `Option<Option<WlSurface>>` (outer None = inactive)
- Focus: `SeatHandler::KeyboardFocus = KeyboardFocusTarget` enum (Window/Layer/Popup) in `focus.rs` â€” lets smithay's per-variant `KeyboardTarget` impl handle protocol specifics. Single-policy helper `EmskinState::auto_focus_new_window(window, window_id)` is the one entry `xdg_shell::new_toplevel` calls â€” respects `prefix_saved_focus`. On window destroy, fallback to `emacs_focus_target()` if `current_focus().is_none()`; Emacs's buffer MRU drives further recovery via `set_focus` IPC. X clients arrive through xwayland-satellite and present as ordinary Wayland clients, so there is no dedicated X11 focus plumbing in emskin itself â€” satellite handles ICCCM `SetInputFocus` / `WM_TAKE_FOCUS` and EWMH focus-state bits internally before translating to `wl_keyboard.enter`
- Embedded toplevel configure: `ipc_set_geometry` sets all four `TiledLeft/Right/Top/Bottom` states so terminal emulators (foot) fill the exact configured size with padding instead of rounding to cell boundaries
- Window destroy (Elisp): `emskin--on-window-destroyed` does `delete-window` (if multi-window) then `kill-buffer`, then sends `set_focus` for `(window-buffer (selected-window))`. Use `window-buffer (selected-window)` not `current-buffer` after `kill-buffer` â€” the latter is unreliable
- IME input: text_input_v3 bridges host IME (via winit Ime events) to embedded Wayland clients (Chrome). GTK/Qt apps use their own IM module (fcitx5-gtk via DBus) and are unaffected â€” `set_ime_allowed` is only enabled when the focused client has bound text_input_v3. Smithay patches required: expose `WinitEvent::Ime`, remove `has_instance()` guard in text_input dispatch, add `cursor_rectangle()` accessor
- `AppWindow::wl_surface()` returns the toplevel's `WlSurface`.
- XWayland: emskin does not embed its own X server. `xwayland-satellite` runs as a dedicated external process managed by `crates/emskin/src/xwayland_satellite/` with a niri-style on-demand supervisor: emskin pre-binds `/tmp/.X11-unix/X<N>` and the Linux abstract socket, then `arm()`s calloop `Generic` sources on both fds; the first X client connect triggers `on_socket_connect()` which spawns the satellite in a dedicated thread, and `ToMain::Rearm` via a `calloop::channel` re-installs the watch after a crash. From emskin's viewpoint the satellite is just another Wayland client â€” X-specific focus, cursor, clipboard, and fullscreen policy all live inside satellite, not in emskin.
- Elisp: per-app buffers must use `generate-new-buffer` (not `get-buffer-create`) â€” two same-titled windows (two xterms, two firefox) would otherwise share a buffer and `setq-local emskin--window-id` would clobber the earlier window's id. `rename-buffer ... t` already handles title_changed collisions
- Reference compositors when unclear about focus/input semantics: anvil (`anvil/src/focus.rs` inside the smithay checkout) for the `KeyboardFocusTarget` / `PointerFocusTarget` shape; niri (`~/study/rust/source/niri/src/utils/xwayland/`) for the on-demand satellite supervisor pattern this module is ported from. Cite concrete file:line before asserting behavior.
- Elisp auto-connect: `emskin-maybe-auto-connect` must NOT be gated on `(featurep 'pgtk)` â€” gtk3 Emacs running under xwayland-satellite also needs IPC. Gate only on socket file existence.
- `EmskinState::output_fullscreen_geo()` â€” shared helper for outputâ†’modeâ†’scaleâ†’logical fullscreen geometry, used by resize logic.
- grabs/ directory is placeholder code for future move/resize support
- Workspace model: each Emacs frame = one workspace. Active workspace's state lives in `self.space` + `self.emacs_surface`; inactive workspaces stored in `inactive_workspaces: HashMap<u64, Workspace>`. Switching = swap (std::mem::take). `sync-frame` only processes the hook-triggering frame when its workspace-id matches `active-workspace-id` â€” eliminates race conditions during workspace switches
- App migration: Emacs drives migration via IPC â€” compositor does NOT auto-migrate on workspace switch (doesn't know which apps are in which Emacs frame). `ipc_set_geometry` calls `migrate_app_to_active()` which resets geometry/pending_geometry/pending_since to None so the next set_geometry maps immediately (otherwise pending path deadlocks: app needs frame callbacks but isn't in any Space)
- Elisp workspace switch: `emskin--resync-workspace` clears change detection then delegates to `sync-frame`. `other-frame` advice sends `switch_workspace` IPC but does NOT resync â€” resync is handled by `on-workspace-switched` when the compositor confirms (avoids stale `active-workspace-id`). `WorkspaceSwitched` IPC is sent inside `switch_workspace()` BEFORE `keyboard.set_focus` so Emacs updates state before GTK focus-change hooks fire
- Mirror vs migration: mirrors are for simultaneous visibility (future side-by-side); migration is for workspace switching (one visible at a time). `sync-frame` only processes the active workspace's frame, so invisible workspace frames don't trigger mirror creation
- Child frame detection: pgtk child frames (posframe, company-posframe) also create xdg_toplevel from same Wayland client. Deferred to idle callback via `pending_emacs_toplevels` â€” check `ToplevelSurface::parent()` after dispatch_clients (GTK batches get_toplevel + set_parent in same Wayland message). Parent present = child frame (stays in current space); absent = real frame (creates workspace)
- Workspace bar: external binary `crates/emskin-bar/`, auto-spawned via `--bar=auto|none|<path>` CLI flag (default `auto`). Speaks `zwlr-layer-shell-v1` + `ext-workspace-v1`, never emskin IPC. Visibility: â‰Ą 2 workspaces â†’ layer surface mapped with `exclusive_zone=28`; compositor's `LayerMap::non_exclusive_zone()` shrinks accordingly and `relayout_emacs()` (state.rs) sends new SurfaceSize to Emacs. No `bar_height()` / `bar_enabled` in the compositor â€” any layer-shell client anchored top/bottom works the same way
- IPC Y translation: `ipc_set_geometry` / `ipc_add_mirror` / `ipc_update_mirror_geometry` add `emacs_geometry().loc.y` (the current non-exclusive-zone origin) to incoming Emacs-relative `y`. When no bar is mapped this is 0; when the bar is mapped it equals the bar's exclusive zone height
- ext-workspace-v1 protocol: `protocols/workspace.rs` â€” diff-based refresh model, action queue for client requests. Compositor is the single source of truth; IPC and protocol operate on same workspace state
- IPC extensions: Emacsâ†’compositor: `switch_workspace`. Compositorâ†’Emacs: `workspace_created`, `workspace_switched`, `workspace_destroyed`
- EmskinState sub-structs: `wl: WaylandState` (17 smithay protocol fields), `focus: FocusState` (prefix_saved_focus, text_input_focus, pending_ime_allowed, layer_saved_focus), `selection: SelectionState` (clipboard, origins, cached mimes). Handler impls access via `self.wl.compositor_state` etc.
- ClipboardBackend trait: lives in the sibling `emskin-clipboard` crate (zero smithay deps). Three backends â€” `ClipboardProxy` (data-control, `data_control.rs`), `WlDataDeviceProxy` (wl_data_device fallback, `wl_data_device.rs`), `X11ClipboardProxy` (`x11.rs`) â€” all impl the trait. State field: `Option<Box<dyn emskin_clipboard::ClipboardBackend>>`. Constructed via `emskin_clipboard::init(&[BackendHint::DataControl, BackendHint::wl_data_device(ptr)?, BackendHint::X11])` in `main.rs`, which walks the fallback chain and returns the first backend that handshakes. `BackendHint::wl_data_device` is the only unsafe constructor (takes a foreign `*mut wl_display`, winit's in our case). Event loop registration branches on `ClipboardBackend::driver()`: `Driver::OwnedFd` backends (data-control, X11) register their fd with calloop in `main::register_clipboard_source`; `Driver::Piggyback` (wl_data_device) has no owned fd, so `tick.rs` drains via `dispatch()` every tick.
- `emskin-clipboard` uses independent `SelectionKind { Clipboard, Primary }` to stay smithay-free. The bridge layer (`crates/emskin/src/clipboard_bridge.rs`) converts to/from smithay's `SelectionTarget` via two extension traits (`SelectionTargetExt::to_kind` + `SelectionKindExt::to_target`) â€” orphan rules forbid `impl From` here since all three types are foreign to emskin, so extension traits (local to emskin) are the idiomatic escape hatch. Callsites in `handlers/mod.rs` `use crate::clipboard_bridge::SelectionTargetExt;` and call `target.to_kind()`.
- X11-only async completion: the `X11ClipboardProxy` backend, on host paste requests, emits `HostSendRequest { completion: Some(AsyncCompletion { id, read_fd }) }` so the bridge layer can drain the pipe via calloop and call back into `ClipboardBackend::complete_outgoing(id, data)` for the backend to send `SelectionNotify`. Wayland backends emit `completion: None` â€” the default `complete_outgoing` impl is a no-op. Error path in the pipe reader still calls `complete_outgoing` with whatever was drained so `outgoing_requests[id]` never leaks and the X11 requestor always gets a reply.
- IpcRect: shared `{x, y, w, h}` struct with `#[serde(flatten)]` â€” replaces repeated bare fields in SetGeometry, AddMirror, UpdateMirrorGeometry, SkeletonClicked, SkeletonRect
- Module layout: `lib.rs` (library entry), `tick.rs` (event loop body), `ipc/dispatch.rs` (IPC message handlers), `clipboard_bridge.rs` (smithay glue for the `emskin-clipboard` crate), `mirror_render.rs` (mirror texture rendering)
- Overlays live in the `effect-plugins` crate (measure / skeleton / splash). The host keeps `Rc<RefCell<T>>` handles and registers them into `state.effect_chain` via `effect_core::EffectHandle`. Typed pub methods drive state (`set_enabled`, `set_rects`, `dismiss`, `click_at`); the `Effect` trait itself is purely visual. See `../effect-plugins/CLAUDE.md`.
- Render loop: `winit.rs::render_frame` builds `EffectCtx { canvas = state.emacs_geometry() }` + extras (software cursor, layer-shell, mirrors) and calls `effect_core::render_workspace` once. `canvas` is the single coordinate source â€” effects paint inside it, never using output-absolute coords. Effect-core owns the smithay `render_output` call.

## Key Gotchas
- smithay winit backend defaults to 10-10-10-2 pixel format (2-bit alpha) â€” breaks GTK semi-transparent UI. Fixed by prioritizing 8-bit in smithay's `backend/winit/mod.rs`
- GPU readback on winit backend: `ExportMem::copy_framebuffer` must run inside the `backend.bind()` block while the EGL surface is still current, but `map_texture` must run AFTER `backend.submit()` â€” `map_texture` internally `make_current`s without a draw surface, detaching the winit EGL surface and breaking the next `eglSwapBuffers` with `BAD_SURFACE`. See `capture.rs` + `recording.rs` for the split-across-submit pattern
- For capture / mirror paths, trust `backend.window_size()` for physical framebuffer dimensions, NOT `output.current_mode().size` â€” on fractional-scale resizes (e.g. KDE 1.624Ă—) winit resizes its EGL surface immediately but the output's mode is re-synced on the next render tick; reading mode.size at capture time gives a lagged size and produces stride-mismatched pixel data
- winit `scale_factor()` returns 1.0 at init time; real scale arrives later via `ScaleFactorChanged` â†’ `WinitEvent::Resized { scale_factor }`
- Use `Scale::Fractional(scale_factor)` not `Scale::Integer(ceil)` to match host compositor's actual DPI
- `render_scale` in `render_output()` should be 1.0 (smallvil pattern); smithay handles client buffer_scale internally
- `Transform::Flipped180` is required for correct orientation with the winit EGL backend
- Use smithay's type-safe geometry: `size.to_f64().to_logical(scale).to_i32_round()` instead of manual arithmetic
- GTK3 Emacs does NOT support xdg-decoration protocol â€” setting `Fullscreen` state on the toplevel is what actually hides CSD titlebar/borders
- GTK4/GTK3 will send `unmaximize_request`/`unfullscreen_request` immediately on connect if those states are set in initial configure â€” must ignore these for single-window compositor
- Host keyboard layout: smithay winit backend does NOT expose the host's keymap. Use `wayland-client` to separately connect, receive `wl_keyboard.keymap`, then `KeyboardHandle::set_keymap_from_string()` â€” env vars (`XKB_DEFAULT_*`) are unreliable on KDE Wayland
- pgtk Emacs: `frame-geometry` returns 0 for `menu-bar-size` (GTK external menu-bar architectural limitation, not a bug). Compute exact bar height via compositor IPC: `offset = surface_height - frame-pixel-height`
- `window-pixel-edges` is relative to native frame (excludes external menu-bar/toolbar); `window-body-pixel-edges` bottom = top of mode-line
- embedded app windows must be mapped to space at 1Ă—1 in `new_toplevel` (otherwise on_commit and initial configure don't fire); actual size arrives via `set_geometry` IPC
- Host resize must only resize the Emacs surface; embedded app window sizes are controlled by Emacs via IPC
- Mirror rendering: `TextureRenderElement` position is Physical coords â€” must use `output.current_scale().fractional_scale()` for logicalâ†’physical conversion, NOT hardcode 1.0
- Mirror rendering must walk the full `wl_subsurface` tree via `with_surface_tree_downward` â€” GTK/Firefox paint content onto subsurface children, so reading only the root surface yields an empty mirror
- Mirror rendering: call `import_surface_tree` once per layer, then walk each layer's subsurface tree *once* (not per mirror) and scale the collected snapshots â€” avoids O(mirrors Ă— tree) traversals in the render hot path
- Mirror element Id must be `Id::from_wayland_resource(surface).namespaced(view_id as usize)` â€” same surface in different mirrors needs distinct Ids or the damage tracker collapses them. `render_elements_from_surface_tree` cannot replace the manual walk because its Id is hardcoded to `from_wayland_resource(surface)` with no namespace hook
- Mirror rendering must subtract `window.geometry().loc` (and `popup.geometry().loc` for popups) from the render origin â€” GTK/Chrome put CSD shadow padding in the buffer and use `xdg_surface.set_window_geometry` to mark where the visible window actually starts. Smithay's `Space::render_location()` does `space_loc - element.geometry().loc` automatically; custom mirror paths must match or visible content gets pushed inward by the shadow amount. Precompute this into `SurfaceLayer::render_offset` (popup offset minus geometry offset) so per-layer walks don't redo the math
- Mirror rendering: `TextureRenderElement` needs `buffer_scale`, `buffer_transform`, and viewport `src` from `RendererSurfaceState` â€” otherwise size is wrong under fractional scaling
- Mirror input: `surface_under()` must check mirrors BEFORE space â€” Emacs is fullscreen and `element_under()` always hits it first, blocking mirror detection
- Mirror input: pointer `under_position` for mirrors needs offset compensation (`pos - mapped_pos`) so smithay computes correct surface-local coords
- Mirror input: `surface_under()` for mirrors must compensate `window.geometry().loc` â€” same CSD shadow offset that the space path handles via `render_location = space_loc - geometry.loc`. Add `wg` to `local` point and subtract `wg` from `surface_global` in the return mapping, otherwise cursor hits shadow area instead of visible content
- Mirror scaling: aspect-fit with top-left alignment; coordinate mapping in `mirror_under` uses `rel.downscale(ratio)` to map mirrorâ†’source; `AppManager::aspect_fit_ratio()` returns None for zero-size to prevent NaN
- `render_output`'s second type param is the custom_elements type (not space element type); `render_scale` (value 1.0) is actually the `alpha` parameter
- `render_elements!` macro cannot parse associated-type bounds (`Renderer<TextureId = GlesTexture>`) â€” define a blanket helper trait as workaround. The `CustomElement` enum + `EmskinRenderer` trait live in `effect-core::element` now
- Custom overlays: `SolidColorRenderElement` for shapes, `MemoryRenderBuffer` + bitmap font for text. `CommitCounter` must be stored in struct and incremented on change â€” `default()` every frame defeats damage tracking. Helpers `paint_buffer` / `draw_text_onto` live in `effect-core::render`
- Elisp `defcustom` with `:set` that references later-defined vars: use `:initialize #'custom-initialize-default` + `bound-and-true-p` to avoid void-variable at load time
- IME: registering `TextInputManagerState` causes fcitx5-gtk to switch from DBus to text_input_v3 path â€” must dynamically toggle `set_ime_allowed` per focused client (only enable when client has bound text_input_v3, check via `with_focused_text_input`)
- IME: smithay's keyboard.rs gates `text_input.enter()/leave()` behind `input_method.has_instance()` â€” without input_method, must manually call enter/leave in `SeatHandler::focus_changed` with temporary focus swap to send leave to the correct old client
- IME: `pending_ime_allowed: Option<bool>` pattern â€” `focus_changed` cannot access winit backend, so deferred to `apply_pending_state` (same pattern as `pending_fullscreen`/`pending_maximize`)
- Elisp: use `window-body-pixel-edges` for embedded app geometry (excludes fringes/margins/header-line/mode-line). Set buffer-local `left-fringe-width`, `right-fringe-width`, `left-margin-width`, `right-margin-width` to 0 and `cursor-type` to nil for EAF buffers
- Elisp: `set-window-scroll-bars` is non-persistent across buffer switches â€” re-apply in `emskin--sync-frame` unconditionally for EAF windows
- Elisp skeleton: guard bar height fallback with `frame-parameter 'menu-bar-lines/tool-bar-lines/tab-bar-lines` â€” pgtk fallback incorrectly derives non-zero heights for disabled bars without this check
- Popup input: clicking a popup surface must NOT change keyboard focus if the popup belongs to the same Wayland client as the current focus â€” `wl_keyboard.leave` on the parent toplevel causes Firefox/Chrome to dismiss the popup before processing the button event. Use `same_client_as()` guard before `keyboard.set_focus()`
- Popup input: browsers (Firefox, Chrome) may open menus as `xdg_popup` WITHOUT requesting `xdg_popup.grab` â€” the compositor must handle ungrabbed popups via the normal pointer focus path (no `PopupPointerGrab`)
- Clipboard under satellite: X clients reach emskin as ordinary Wayland clients through `xwayland-satellite`'s internal translator. Their selections arrive on `wl_data_device` just like pgtk Emacs, so `SelectionHandler::new_selection` is the single entry point â€” there is no separate XwmHandler path and no `X11Wm::new_selection` replay to worry about. `SelectionOrigin { Wayland, Host }` (the `X11` variant was removed with the smithay X11Wm path).
- Clipboard startup guard: use `!self.ipc.is_connected()` instead of per-target bool flags â€” GTK clipboard init happens before emskin.el connects, real user copies happen after.
- Cursor under satellite: `xwayland-satellite` forwards X11 cursor changes to emskin via `wp_cursor_shape_v1` / `wl_pointer.set_cursor` like any other Wayland client, so `SeatHandler::cursor_image` works uniformly for pgtk and gtk3 Emacs. No XFixes tracking in emskin itself.
- Layer shell (wlr-layer-shell): uses smithay's `LayerMap` + `DesktopLayerSurface` (not manual Vec). `layer_map_for_output` returns MutexGuard â€” collect data and drop guard before calling renderer or keyboard.set_focus. Frame callbacks: collect layers, drop guard, then send_frame
- Layer shell keyboard focus timing: `new_layer_surface` fires on `get_layer_surface` (BEFORE initial commit) â€” cached_state has no keyboard_interactivity yet. Must defer focus to compositor commit handler where `can_receive_keyboard_focus()` works correctly
- Layer shell destroy: only reclaim focus if `keyboard.current_focus() == destroyed surface` â€” unconditional fallback steals focus from other active windows
- Code language: all comments, tracing logs, and doc comments must be in English. No Chinese in Rust source code
- Workspace switch must reset: `prefix_saved_focus`, `text_input_focus`, `pending_ime_allowed`, `skeleton`, `cursor_status` (if Surface), pointer focus (send motion with None). Without this, stale references to old workspace surfaces cause events sent to wrong clients
- Focus feedback loop on workspace switch: Wayland keyboard.enter and IPC arrive via different sockets with non-deterministic ordering. Fix: (1) `sync-frame` only processes the hook-triggering frame, checking frame's own ws-id against `active-workspace-id` â€” stale hooks for the wrong frame are rejected; (2) `WorkspaceSwitched` IPC sent before `keyboard.set_focus` inside `switch_workspace()`; (3) `emskin--workspace-switch-suppressed` flag prevents `after-focus-change` feedback loop
- New Emacs frame fullscreen: must send configure with `Fullscreen` state + output size in `new_toplevel` (not defer to `handle_surface_commit`). Otherwise GTK renders first frame with CSD decorations at default size
- `set-window-scroll-bars`/`set-window-fringes`/`set-window-margins` are non-persistent across frame creation â€” unconditionally reset to 0 in `sync-frame` for EAF windows
- `emskin--last-focused-wid` must be reset to `'unset` on workspace switch â€” otherwise `sync-focus` skips re-sending `set_focus` when the same app was focused before the switch (compositor already reset focus to Emacs during `switch_workspace`)
- `other-frame` (C-x 5 o): advised `:around` to send `switch_workspace` IPC before calling original. Does NOT resync after â€” `on-workspace-switched` handles it when compositor confirms
- Layer-shell non-exclusive zone changes (bar map/unmap/reconfigure) trigger `EmskinState::relayout_emacs()` â€” resizes every Emacs frame (active + inactive workspaces) to `usable_area()` and broadcasts `SurfaceSize` IPC. Hooked in three places: `handlers/layer_shell.rs::{new_layer_surface, layer_destroyed}` and `handlers/compositor.rs::commit()` (when `LayerMap::arrange()` returns `changed = true`)
- Layer-shell relayout gating: `LayerMap::arrange()` returns true on any layout change including non-exclusive overlays (rofi/zofi) repositioning. Only trigger `relayout_emacs()` when `map.non_exclusive_zone()` differs before vs. after arrange; gating on `arrange()`'s bool causes every launcher keystroke to resize every Emacs frame (visible as white flashes in embedded apps)
- Pointer constraints activation: smithay does NOT auto-activate `zwp_pointer_constraints_v1` constraints inside `PointerHandle::motion` â€” the compositor must explicitly call `PointerConstraintRef::activate` both in `new_constraint` (when the target surface is already focused) AND on pointer-enter into a surface that has a pending constraint. Missing either site silently breaks pointer lock for games
- Relative pointer delta synthesis: smithay's winit backend only emits `PointerMotionAbsolute` â€” there is no raw-motion source. To feed `zwp_relative_pointer_v1` (required by Minecraft, Blender, browser Pointer Lock), store `last_pointer_raw_loc: Option<Point<f64, Logical>>` on `EmskinState` and diff consecutive host-reported absolutes to produce the delta. This delta is independent of the compositor's own `pointer.current_location()` (which freezes under a pointer lock), so it remains correct while locked. `delta_unaccel` uses the same value â€” we have no unaccelerated source from winit, which is a known minor sensitivity tradeoff vs. a libinput backend
- Pointer motion under constraint: when a constraint is active and Locked, skip `pointer.motion()` entirely (only emit `pointer.relative_motion()` + `pointer.frame()`) â€” otherwise the compositor cursor slides while the client thinks it's frozen. When Confined, check that the new position stays within surface + region before calling `motion()`, otherwise suppress the motion
- Wayland child processes must have `WAYLAND_DISPLAY` explicitly set to `state.socket_name` â€” default env inheritance points at the host compositor. Applies to both `spawn_child` (Emacs) and `spawn_bar` (emskin-bar). Symptom of missing env: child dies with `NoCompositor` / global bind errors and shows up as `<defunct>` until emskin exits
- Buffer-space coordinates: anything writing into a `MemoryRenderBuffer` uses `Point<i32, Buffer>` / `Size<i32, Buffer>` smithay markers, not raw `i32` â€” prevents mixing Logical/Physical/Buffer in pixel-write hot paths (see `effect_plugins::measure::draw_text`, `effect_core::render::paint_buffer`)
- Release workflow: `crates/emskin/Cargo.toml` `version` MUST equal the git tag minus the `v` prefix â€” `cargo aur` bakes `pkgver` into the PKGBUILD source URL (`releases/download/v$pkgver/...`). Mismatch â†’ AUR PKGBUILD downloads a stale/missing tarball. Bump Cargo.toml version before tagging. Note: cargo-aur 0.x does not support `version.workspace = true`, so this crate keeps the literal value
- AUR publish action pinned to `KSXGitHub/github-actions-deploy-aur@v4.1.2` or newer â€” v4.1.1 breaks on recent Arch `util-linux` with `bash: --command: invalid option`
- Verifying AUR state: `https://aur.archlinux.org/cgit/aur.git/plain/PKGBUILD?h=<pkg>` is cached â€” use `git clone https://aur.archlinux.org/<pkg>.git` to read the actual repo head
- pgtk text_input_v3 is unreliable for caret position: `GTK_IM_MODULE=fcitx` intercepts the GTK IM context and pgtk never sends `zwp_text_input_v3.enable`, so `set_cursor_rectangle` doesn't fire.  Even `GTK_IM_MODULE=wayland` with `im-wayland.so` loaded didn't trigger `enable` in testing.  For caret tracking, use elisp IPC (`post-command-hook` + `pos-visible-in-window-p`, mirroring holo-layer) â€” see `elisp/emskin-jelly.el`

## Testing

- E2E entry: `cargo build -p emez && cargo test -p emskin`. **Two steps is a cargo-stable limitation**, not a bug: emez is a binary crate and cargo stable has no bindeps (`-Z bindeps` / [RFC 3028](https://rust-lang.github.io/rfcs/3028-cargo-binary-dependencies.html) is nightly-only), so there's no first-class way to say "build this binary before my tests". The harness falls back to runtime discovery via `find_emez_binary()` (scans `target/{debug,release}/emez`); if the binary isn't there it panics with a clear message pointing at the build step. A future `cargo-xtask` wrapper could collapse this to one command, but for current project scale it's overengineering. Tests run in **parallel**; the harness pre-allocates a unique X DISPLAY number for every emez-host and emskin instance through a process-wide reservation pool (`DisplaySlot`) and passes them via `--xwayland-display`, so smithay's XWayland bootstrap never races. Each test also spawns its **own** private host compositor (`NestedHost::wayland()` â†’ **emez** headless, our smithay-based sister crate in `crates/emez/`; `NestedHost::x11()` â†’ Xvfb) into a dedicated `XDG_RUNTIME_DIR`.
- Lifecycle: `WaylandHost::drop` / `Compositor::drop` send SIGTERM (not `Child::kill`, which is SIGKILL) so emez runs its full Rust Drop chain and kills the Xwayland child. emez registers a SIGTERM handler (signal-hook) that calls `LoopSignal::stop()` + `LoopSignal::wakeup()` to break out of `event_loop.run` immediately. Signal-driven exits occasionally truncate the Drop chain, so emez and harness both remove `/tmp/.X11-unix/X<N>` + `/tmp/.X<N>-lock` as a belt-and-braces step.

- IPC wire-format quirk: `#[serde(rename_all = "snake_case")]` on `OutgoingMessage` turns `XWaylandReady` into `x_wayland_ready` (not `xwayland_ready`). Matchers in tests/elisp must use the underscore form. Under satellite, `XWaylandReady` fires as soon as emskin has pre-bound the X11 sockets â€” the satellite process itself may still be unspawned at that point because it's on-demand.
- `IpcServer::send` drains its write buffer synchronously on EAGAIN (temporarily switches fd to blocking). calloop IPC source is READ-only, so without this drain, late async messages (e.g. `XWaylandReady`) can stall indefinitely. Production is masked by Emacs's constant incoming traffic driving tick.
- xclip gotchas: (1) default `-loops=0` means xclip **never exits** â€” spawn and don't wait. (2) If stderr is piped, xclip's forked child inherits the pipe's write end and `wait_with_output` deadlocks on EOF â€” use `Stdio::null()`.
- `SelectionOrigin` has two variants: `Wayland` / `Host`. `inject_host_selection` (clipboard_bridge.rs) sets it to `Host`; `send_selection` branches to route paste fds (wayland client â†’ data_device, compositor-injected host selection â†’ `ClipboardBackend::receive_from_host`). X clients under satellite are wayland clients from emskin's point of view, so they fall into the `Wayland` branch automatically.
- External host sync still goes through whichever `emskin-clipboard` backend `init()` picked (data-control â†’ wl_data_device â†’ X11 fallback chain).
- calloop 0.14 `LoopSignal::stop()` only sets a flag; it does NOT wake the event loop. Signal handlers must call `stop(); wakeup();` as a pair, otherwise `event_loop.run` stays blocked in epoll_wait and never observes the flag.
- `std::process::Child::kill()` sends SIGKILL and skips the Rust Drop chain. The harness uses `graceful_kill` (SIGTERM + 1.5s wait + SIGKILL fallback) on both emez and emskin so neither orphans its embedded XWayland (emez still uses smithay's built-in X11Wm internally for the external-X test cells) nor leaks an `xwayland-satellite` child.
- Display-number allocation: emskin's satellite supervisor scans `/tmp/.X<N>-lock` from 0 for a free slot â€” parallel test spawns race the same way smithay's old `XWayland::spawn(None, ...)` did. The harness still uses `DisplaySlot` â€” a process-wide `Mutex<Vec<u32>>` + fs-existence check + RAII release â€” and passes the reservation via `--xwayland-display <N>` so every emez and every emskin instance claims a disjoint number.
- X socket/lock (`/tmp/.X11-unix/X<N>` + `/tmp/.X<N>-lock`) cleanup is owned by the `Unlink` RAII guards in `xwayland_satellite::sockets::X11Sockets`. Signal-driven exits can truncate the Rust Drop chain, so both emez (`main.rs` after `event_loop.run`) and `Compositor::drop` (`tests/common/mod.rs`) also `fs::remove_file` the pair as a belt-and-braces step.
